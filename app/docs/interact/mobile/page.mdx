## The use case

Long-running agents don't need you at your desk. Start a coding task, head to lunch,
check progress from your phone. Your Omnigent hits a question. Answer it from the coffee
line. Want to review what it produced? Browse the files right there.

Everything syncs in real time across devices.

## How to access

`localhost:6767` is only reachable from the machine running the server. To access
from your phone:

**Same network.** Use your machine's local IP instead of localhost (e.g.,
`http://192.168.x.x:6767`). Your phone needs to be on the same Wi-Fi network.

**From anywhere.** Deploy the Omnigent server beyond localhost. See
[the deployment docs](/docs/deploy/overview) for options.

No app to install. Just open the URL in your mobile browser. An optional
[native Android shell](#native-android-app) is also in development for people who
want OS notifications and a home-screen app icon.

## What works on mobile

The full Web UI, touch-optimized: chat, monitor sub-agents, browse files, view code changes,
and leave inline comments.

## Native Android app

Alongside the mobile browser, Omnigent has a native Android shell — a thin
`WebView` wrapper that loads your server's web UI instead of shipping a separate
copy of it. Like the [desktop app](/docs/interact/desktop) and the iOS shell, it
adds OS-level features a browser tab can't provide while running the exact same
UI. It's an optional client: the mobile browser remains the zero-install way in.

On first launch you enter your **Server URL** (e.g.
`https://omnigent.example.com`); the app loads that server's web UI directly and
remembers recently used servers.

### What the shell adds over the browser

- **OS notifications** when a session needs your attention, with tap routing
  back into the relevant session.
- **A best-effort app badge.** Android has no universal numeric badge, so the
  shell relies on the notification dot as the guaranteed surface (some launchers
  also show a count).
- **File downloads**, including `blob:` and `data:` exports that the mobile
  browser and iOS shell can't save.
- **File uploads** from `<input type="file">`, and **microphone capture** for
  voice input (the app requests microphone access for your pinned server only).
- **Edge-to-edge inset handling** so content lays out correctly around the
  status bar and navigation area.

The sidebar, server switcher, and chat/terminal bar are rendered by the web UI
in-page (the same as a browser tab), so they work without any native chrome.

### Status and install

The Android shell is early: it builds cleanly but has not yet been verified on a
device, and it isn't published to an app store. Because Google Play restricts
"WebView of a website" apps, the initial distribution channel is a direct
APK / F-Droid rather than the Play Store.
