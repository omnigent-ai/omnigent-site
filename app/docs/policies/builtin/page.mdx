# Builtin Policies

Omnigent ships with policies for common guardrails, organized into two categories:
**Safety** and **Cost Control**. Your Omnigent can apply any of
these by name when you [ask it to add a policy](/docs/policies/overview#adding-a-policy), or you
can reference them in YAML by their full path in the `handler` field.

All builtin policies live under `omnigent.policies.builtins`.

## Safety

| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ask_on_os_tools`            | ASKs before any file or shell operation.                                                                                                                                                                   | None                                                                                                                                                                                              |
| `block_skills`               | Prevents specific skills from loading.                                                                                                                                                                     | `blocked` (string[], required)                                                                                                                                                                    |
| `block_working_dir_changes`  | Blocks shell commands that change the working directory.                                                                                                                                                   | `block_cd` (bool), `block_worktree` (bool), `allowed_dirs` (string[]), `action` (`"deny"` or `"ask"`)                                                                                             |
| `cel_policy`                 | Write custom policy logic using CEL (Common Expression Language), a safe, non-Turing-complete expression language.                                                                                         | `expression` (CEL expression string), `reason` (deny message)                                                                                                                                     |
| `deny_pii_in_llm_request`    | Scans outgoing messages for PII and blocks or flags them.                                                                                                                                                  | `pii_types` (string[]), `action` (`"DENY"` or `"ASK"`)                                                                                                                                            |
| `enforce_sandbox`            | Forces a sandbox configuration on agent start.                                                                                                                                                             | `sandbox_type`, `allow_network`, `write_paths`, `read_paths`                                                                                                                                      |

| `gmail_policy`               | Controls Gmail. Defaults to read + draft, no send.                                                                                                                                                         | `allow_read`, `allow_send`, `allow_drafts`                                                                                                                                                        |
| `max_tool_calls_per_session` | DENYs after a total tool-call limit is reached.                                                                                                                                                            | `limit` (int, default `100`)                                                                                                                                                                      |
| `prompt_policy`              | Evaluate policy decisions using an LLM. The policy sends the event context to a model and interprets the response as ALLOW/ASK/DENY. | `risk_score_policy`          | Accumulate a risk score from tool calls and sensitive data labels. Escalates guarded tools to ASK or DENY once the score exceeds a threshold.                                                              | `threshold` (int), `tool_points` (object mapping tool names to points), `sensitive_labels` (object mapping labels to points), `guarded_tools` (string[]), `escalate_action` (`"ASK"` or `"DENY"`) |

## Cost Control

| Policy                            | What it does                                                                                                 | Parameters                                                                |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------- |
| `cost_budget`                     | Tracks cumulative LLM spend per session. ASKs at soft thresholds, blocks expensive models at the hard limit. | `max_cost_usd` (required), `ask_thresholds_usd`, `expensive_models`       |
| `detect_task_switch`              | Uses the server-level LLM to detect when a user starts a new unrelated task and nudges starting a fresh session. | `min_turns` (int), `history_window` (int), `action` (`"ASK"` or `"DENY"`, default `"ASK"`), `classification_prompt` (string) |
| `deny_trivial_to_expensive_model` | Classifies messages as trivial or complex. Routes trivial tasks away from expensive models.                  | `expensive_models` (string[], required), `classification_prompt` (string) |
| `user_daily_cost_budget`          | Same as `cost_budget`, but enforced per-user daily across all sessions.                                      | `max_cost_usd` (required), `ask_thresholds_usd`                           |

### Usage examples

The instinct when AI spend grows is to lower caps and add friction. The more
effective approach is visibility and soft guardrails — so usage can keep
growing without cost growing at the same rate.

#### Make spend visible before you restrict it

Start with soft thresholds only. Engineers see when they're running an
expensive session and can decide whether the task warrants it. No one gets
blocked; habits change on their own.

```yaml
# Omnigent config (policies block)
policies:
  session_visibility:
    type: function
    handler: omnigent.policies.builtins.cost.cost_budget
    factory_params:
      ask_thresholds_usd: [1.0, 5.0]
      max_cost_usd: 999.0 # effectively no hard cap
  daily_visibility:
    type: function
    handler: omnigent.policies.builtins.cost.user_daily_cost_budget
    factory_params:
      ask_thresholds_usd: [10.0, 25.0]
      max_cost_usd: 999.0
```

This gives you real data on where spend is concentrated before you decide
where to add guardrails.

#### Server-wide team policy

A reasonable starting point for a team deployment: per-user daily visibility
at $25, a soft check at $50. Engineers can still use any model for any
task — they just get asked before spending more than $25 in a day.

```yaml
# config.yaml
policies:
  daily_budget:
    type: function
    function:
      path: omnigent.policies.builtins.cost.user_daily_cost_budget
      arguments:
        ask_thresholds_usd: [25.0, 50.0]
        max_cost_usd: 100.0
```

```
omnigent server -c config.yaml
```

The daily cap exists to surface sessions that are genuinely off the rails,
not to penalize productive use.

### `detect_task_switch`

This policy helps control cost by keeping an agent's working context lean. The goal isn't fewer
tokens _used_ but fewer tokens _wasted_ — a session whose context is filled with
stale tool results from a previous, unrelated task degrades quality without adding
value. The recommended response is to start a fresh session for the new task
rather than compacting or summarizing in place.

The policy fires on user `request` events. It keeps a rolling window of recent
user messages in session state and, once enough prior messages have accumulated
(`min_turns`), asks the server-level LLM to classify the latest message. On a
`TASK_SWITCH` verdict it returns `action` with a message recommending a fresh
session; on a `CONTINUATION` verdict it records the message and lets the request
through.

- **`min_turns`** — number of prior user messages to accumulate before the
  classifier starts firing. Set to `0` to classify from the very first message.
- **`history_window`** — maximum number of recent user messages kept as prior
  context for the classifier; older messages are dropped as the window slides.
- **`action`** — response when a task switch is detected. `"ASK"` (default)
  escalates to the user; `"DENY"` blocks the request outright. `"ASK"` is the
  default because a false-positive task-switch classification is more harmful
  than a false negative.
- **`classification_prompt`** — system prompt for the classifier. It must instruct
  the model to return `{"verdict": "CONTINUATION"|"TASK_SWITCH"}`; the output
  schema is enforced via structured output regardless.

This policy requires the server to have an `llm:` config block. When no LLM client
is available it abstains (fail-open), letting the request through.

<div className="note">

**Note:** `action: "DENY"` is a context-hygiene nudge, not a security control.
User messages are interpolated into the classifier prompt, so a determined user
can craft a message that forces a `CONTINUATION` verdict. Use this policy for
context hygiene, not access control.

</div>

```yaml
# Omnigent config (policies block)
policies:
  keep_context_lean:
    type: function
    handler: omnigent.policies.builtins.context.detect_task_switch
    factory_params:
      action: ASK
```
